### 🔴 A1 — CRITIQUE (CORRIGÉ) : `_get_db` indéfinie

**Constat** : `_get_db` était référencée dans 17 signatures d'endpoint mais n'existait jamais dans `server.py`. Seule `_get_db_optional` était définie. Le module ne chargeait pas — le serveur ne démarrait pas.

```
NameError: name '_get_db' is not defined
  File "server.py", line 325, in <module>
    db: CorpusDB | None = Depends(_get_db),
```

**Présent depuis** : commit `26d1967` (MX-028 — premier endpoint utilisant `_get_db`)
**Fix appliqué** : commit `fix(server): define _get_db — 17 endpoints unreachable` — ajout de `_get_db()` strict qui lève 503 si `corpus.db` absent.

---

### 🟡 A2 — WARN : Guards redondants après le fix

**Constat** : Tous les endpoints utilisant `_get_db` contiennent encore un guard `if db is None: raise HTTPException(503, ...)`. Avec la nouvelle `_get_db` qui lève l'erreur elle-même, ces guards ne s'exécuteront jamais.

**Fichier** : `server.py` — environ 15 occurrences
**Impact** : Code mort, mais inoffensif.
**Action recommandée** : Nettoyer les guards `if db is None:` dans les endpoints qui utilisent `_get_db` (pas `_get_db_optional`). Typer `db: CorpusDB` au lieu de `CorpusDB | None`.

### 🟡 B1 — WARN : Foreign Keys manquantes (pas de CASCADE)

**Constat** : La seule FK explicite est `align_links.align_run_id → align_runs`. Les autres relations sont implicites (gérées par code) :

| Table | Colonne | Référence implicite |
|-------|---------|---------------------|
| `segments` | `episode_id` | `episodes(episode_id)` |
| `subtitle_tracks` | `episode_id` | `episodes(episode_id)` |
| `subtitle_cues` | `track_id` | `subtitle_tracks(track_id)` |
| `subtitle_cues` | `episode_id` | `episodes(episode_id)` |
| `align_runs` | `episode_id` | `episodes(episode_id)` |
| `align_links` | `align_run_id` | `align_runs` — FK existe mais sans CASCADE |

**Risque** : Suppression d'un épisode n'entraîne pas de suppression en cascade des segments, cues, runs et links associés → données orphelines silencieuses.

**Mitigation actuelle** : SQLite n'applique les FK que si `PRAGMA foreign_keys = ON` — or le code utilise `PRAGMA foreign_keys = ON` dans `_conn()` ✓. La FK sur `align_links` est donc active. Les autres colonnes n'ont pas de FK, donc aucune protection.

**Action recommandée** : Migration 006 — ajout FK + ON DELETE CASCADE sur les tables enfants. Nécessite SQLite ≥ 3.25 (disponible).

---

### 🔵 B2 — NOTE : Table `runs` vestige

**Constat** : La table `runs` (schema.sql L43) n'est jamais utilisée dans le code (ni en lecture ni en écriture). Elle date du MVP.

**Action** : Documenter comme table archive ou supprimer dans une migration future.

---

### 🔵 B3 — NOTE : FTS5 — `speaker_explicit` absent de l'index

**Constat** : `segments_fts` indexe `(segment_id, episode_id, kind, text)` — le champ `speaker_explicit` n'est pas indexé. La recherche par locuteur dans le concordancier se fait en post-filtre Python (correct).

**Impact** : Pas de recherche FTS native par locuteur. Sur de grands corpus, le post-filtre peut être lent.

**Option future** : Ajouter `speaker_explicit` à l'index FTS, ou créer un index dédié pour une recherche rapide par locuteur.

### 🔴 C1 — CRITIQUE : Export alignements — logique cassée si `pivot_lang ≠ en`

**Constat** : Dans `db_align.get_parallel_concordance()` (db_align.py L576-618), les liens cible sont regroupés sous `target_by_cue_en` indexé par `lnk["cue_id"]`. L'assignation aux colonnes est hardcodée :

```python
text_en = text_pivot if pivot_lang == "en" else ""
text_fr = text_pivot if pivot_lang == "fr" else ""
# ... puis pour les liens target :
if lang == "fr" and cid_t:
    text_fr = cue_fr_by_id.get(cid_t, "")
elif lang == "it" and cid_t:
    text_it = cue_it_by_id.get(cid_t, "")
```

Le schéma d'export assume que :
1. Le pivot est **toujours EN**
2. Les targets sont **toujours FR et IT**

Si `pivot_lang = "fr"` (run segment↔FR direct), alors :
- `text_fr` est rempli depuis le pivot ✓
- Mais la recherche dans `target_by_cue_en` utilise `cue_id_pivot` (FR) comme clé — les target links ayant `role=target` n'existent alors pas dans ce modèle → `text_en` et `text_it` resteront vides

**Impact** : L'export CSV/TSV sera incomplet si le projet utilise un pivot non-anglais.

**Fichier** : `db_align.py` L554-618 + `server.py` L1644-1648

**Action recommandée** : Rendre `get_parallel_concordance()` dynamique — colonnes exportées déduites de `pivot_lang` + langues cibles disponibles dans le run. Le nom de variable `target_by_cue_en` doit devenir `target_by_cue_pivot`.

### 🟡 C2 — WARN : Export alignements écrit sur disque mais le frontend ne lit pas le fichier

**Constat** : `export_alignments()` écrit `out_path.write_text(...)` dans `{project}/exports/` puis retourne `{"path": str(out_path), ...}`. Le frontend `constituerModule.ts` appelle `exportAlignments()` via `apiGet` et reçoit le path, mais **ne lit pas le fichier** — il génère son propre téléchargement client-side depuis les données JSON de la réponse.

**Conséquence** : Les fichiers s'accumulent dans `exports/` sans jamais être lus. L'export côté frontend utilise les données JSON de la réponse API, pas le fichier disque.

**Action recommandée** : Soit retourner le CSV directement en streaming (`StreamingResponse`), soit supprimer l'écriture disque et retourner les données inline.

---

### 🟡 C3 — WARN : Nomenclature `personnage` vs `speaker`

**Constat** : Deux colonnes différentes nommées selon le contexte :
- Export alignements : `personnage` (db_align.py L610 — renommage explicite depuis `speaker_explicit`)
- Export KWIC concordancier (frontend + export_utils) : `speaker`
- DB SQLite : `speaker_explicit`

Trois noms pour le même concept selon le niveau d'abstraction.

**Impact** : Confusion dans les scripts d'analyse externe qui consomment plusieurs exports.

**Action recommandée** : Documenter la convention dans un `SCHEMA.md`, ou harmoniser vers un seul terme (`speaker`).

### 🟡 D1 — WARN : `apiPost()` utilisé pour les requêtes PATCH

**Constat** : Dans `himyc-tauri/src/api.ts`, les appels PATCH utilisent `apiPost(path, body, "PATCH")` au lieu d'une fonction dédiée `apiPatch()`.

```typescript
export async function patchTranscript(...) {
  return apiPost(`/episodes/${episodeId}/sources/transcript`, { clean }, "PATCH");
}
export async function setAlignLinkNote(...) {
  return apiPost(`/alignment_links/${linkId}`, { note }, "PATCH");
}
```

**Impact** : Fonctionnel mais sémantiquement incorrect. Toute future fonction `apiPatch()` devrait être créée, et les appels migrés.
